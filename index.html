0xvouch
v0 · prototype · 2026

AI writes the code.
Reviewers sign the hashes.

A content-addressed lambda service: every function has a unique hash, named developers sign reviews against those hashes, storage is scoped per-space so one reviewed function serves any number of users with their data isolated, and the deploy pipeline rewrites your code into the canonical reviewed primitives — refusing anything that matches a dangerous pattern.

the problem

LLM-generated code is fast but unaudited. Every prompt regenerates the same logic from scratch — same auth flow, same email validator, same off-by-one bug — with no shared notion of what's safe. Trust is by vibes. We can do better.

how it fits together

A hosted service. You write TypeScript (or Claude does, via a skill); the compile pipeline normalizes the AST, rewrites your code against a registry of signed primitives, and refuses anything matching a flagged pattern; the result runs in an isolated sandbox bound to your space's private storage — so the same reviewed function can serve any number of users without their data ever touching each other.

The four sections below explain the core mechanic (steps 01–04). The two after show how it ships (steps 05–06).

01 function = hash

Every function's identity is the hash of its normalized AST. Naming, formatting, and type annotations are stripped before hashing — so a rename produces the same hash, but a behavior change produces a new one.

① source · TypeScript
function isEven(n: number): boolean {
  const result = n % 2 === 0;
  return result;
}
↓parse
② ast · with names & types

Rename n to x? Same hash. Strip the type annotation? Same hash. Change === 0 to !== 0? Different hash. Behavior is the version.

02 reviewers sign hashes

Named developers attach cryptographically signed reviews to specific hashes. Reputation tiers are computed from peer endorsements, audit volume, and retraction rate — think OpenZeppelin for Solidity.

fn:abc1…d3f0auth.hashPassword ✓ FINAL
reviewer @rachel-secops TIER S
signed 2026-04-12 · sig ec1a8f…7b2d
audited scrypt N=16384 r=8 p=1, constant-time compare, salt 16B, no leaks. Safe for password storage.
also signed @oz-eth · @samczsun · @trailofbits-bot

Five signatures from TIER S reviewers = the hash itself becomes TIER S. The signature lives next to the code, not in some external SOC2 PDF.

03 the compiler auto-dedupes

At compile time, every sub-tree in your code is hashed. Matches against the registry are automatically rewritten as imports — you get the canonical reviewed version for free, even if your LLM just regenerated it from scratch.

your code · source
async function register(email, pw) {
  if (!/^\S+@\S+\.\S+$/.test(email))
    throw new Error("invalid email");

  const salt = randomBytes(16);
  const k = scryptSync(pw, salt, 64);
  const stored = `scrypt$16384$8$1$${salt}$${k}`;

  return db.users.insert({ email, stored });
}
fn:def4…1c80validators.email
↳ EXTRACTED TIER S
fn:abc1…d3f0auth.hashPassword
↳ EXTRACTED TIER S
your code · compiled (1 new line, the rest is reviewed)
import { validateEmail } from "fn:def4…1c80";
import { hashPassword }  from "fn:abc1…d3f0";

async function register(email, pw) {
  if (!validateEmail(email))
    throw new Error("invalid email");

  const stored = await hashPassword(pw);

  return db.users.insert({ email, stored });
}

Your 9-line function became 6 lines + 2 imports. The 2 imports pull in TIER S code that's already been audited by named reviewers. Net new unreviewed code: 1 line (the db call). That's all that needs reviewing.

04 dangerous patterns are blocked

Reviewers can also sign negative reviews: "this exact AST is dangerous." Any source whose AST contains a sub-tree matching a flagged hash fails to compile — loudly, with the reviewers and the reason.

your code · with a hidden problem
async function register(email, pw) {
  if (!validateEmail(email))
    throw new Error("invalid email");

  const stored = await hashPassword(pw);

  console.log("registering", email, pw);

  return db.users.insert({ email, stored });
}
fn:bad001…ee04logger.password-leak
✗ FLAGGED 3 × TIER S
scanning sub-trees . . . 47 nodes ✗ match line 7 console.log("registering", email, pw) → matches fn:bad001…ee04 logger.password-leak → flagged by @rachel-secops · @oz-eth · @samczsun all TIER S → reason "logs plaintext password to a destination outside the call boundary" compile aborted. 1 dangerous pattern. fix and retry.

The reviewer didn't need to anticipate your code — they just needed to sign the dangerous pattern once. Every future deploy containing that exact AST shape, anywhere on the platform, is caught.

05 hosted as a service

All of the above happens in our cloud. The function registry lives in a content-addressed object store; user code runs in isolated sandboxes; Claude Code talks to the service through a skill. deploy is the verb you type — under the hood it triggers the compile pipeline from steps 01–04: parse, normalize, hash, dedupe, scan, bundle, run.

$ claude [skill loaded: vouch] > build a small reddit-style forum for my book club vouch: searching registry . . . . . 12 candidate primitives vouch: composing . . . . . . . . . 4 new functions · 247 LoC vouch: compiling . . . . . . . . . 89 LoC after dedupe ✓ 8 extracted ✓ 0 flagged vouch: deploying . . . . . . . . . sandbox provisioned · storage bound vouch: live → https://bookclub.vouch.dev ✓ done. total: 11s.

Same binary is open source. The cloud is the front door; self-host is the escape hatch for when sovereignty matters.

06 storage is a parameter

A reviewed function never owns data — storage is passed in. This means the same hash, the same audited code, can run for a thousand users with their data fully isolated. Content addressing pays off twice: code is shared, storage is not.

one function · signed once · runs anywhere
import { createUser } from "fn:abc1…d3f0";   // TIER S · 5 signatures

function createUser(storage, email, hash) {
  storage.users.insert({ email, passwordHash: hash });
}

isolated

Capability-scoped — each function declares which tables it needs, and the sandbox binds only those, only for that space. The function literally cannot see other tenants' data.
